# 디자인 시스템 분석 가이드라인

이 문서는 디자인 시스템 **수집 결과(통합 표)**를 바탕으로, 구조·의미·중복·접근성 관점에서 품질을 점검하고 개선 항목을 정리하는 절차를 정의한다.

---

## 1. 목적 및 입력 산출물

- **목적**: 수집된 변수·스타일 목록에서 갭·리스크를 식별하고, 후속 수정 항목을 체계적으로 관리한다.
- **선행 조건**: `design_system_collection_guidelines.md` 절차에 따라 통합 표가 갱신되어 있어야 한다.
- **주요 입력**
  - `agent/design/spec/figma-variables-and-styles.md` (통합 표)
  - `agent/design/spec/design_tokens.json`
  - `agent/design/spec/design_principle.json`

---

## 2. 품질 검토 체크리스트

통합 표와 `design_tokens.json`을 나란히 두고 아래를 점검한다.

### 2.1 구조·이름

- [ ] `primary-inactive`처럼 **이름과 실제 의미**(브랜드 vs 중립 트랙)가 어긋나지 않는가.
- [ ] 그림자 색이 `color/` 아래만 있어 **의미가 모호하지 않은가** — 이펙트 네임스페이스·스타일 단일화 검토.
- [ ] `Body` 등 **일반어 스타일명**이 실제로는 플레이스홀더 등 좁은 용도만 쓰이지 않는가.
- [ ] **Figma 심볼·컴포넌트 세트 이름**이 `figma-variables-and-styles.md`·코드·옛 문서의 호칭과 충돌하지 않는가 — **파일 SSOT**로 통일·병합했는가(리네임, 예: `FlatTag`).
- [ ] **variant 속성 값**이 UI 로캘과 같이 **한글** 등으로만 정의되어 있지 않은가. 제품·개발 문서와 **영문 키 병기**가 필요하면 네이밍을 논의한다(예: `Chip`의 `state=기본` / `state=hover`).

### 2.2 의미와 사용처

- [ ] `secondary`가 **브랜드 2차색**인지 **muted 표면**인지 코드 JSON과 Figma가 일치하는가.
- [ ] `on-*` 네이밍이 **실제 배경 토큰과 짝**이 맞는가.
- [ ] 탭 비활성·플레이스홀더가 **같은 색 토큰**을 쓰더라도 **역할 분리**가 필요한지 검토.

### 2.3 중복·SSOT

- [ ] 동일 색이 **여러 변수 경로**로만 존재하면 alias·단일 SSOT로 줄일 수 있는가.
- [ ] 스타일 여러 개가 같은 변수만 가리키는 것은 정상일 수 있으나, **팔레트 변경 시 수정 지점**이 한 곳인가.
- [ ] **`get_design_context`** 참고 코드·스타일 요약에서 **시맨틱 토큰/로컬 스타일과 raw hex**가 **같은 레이어·같은 역할**에 **혼재**하지 않는가.

### 2.4 색상 대비

- [ ] `text-placeholder`, `text-caption`, **작은 글자**·연한 배경 조합을 WebAIM 등으로 샘플 검증.
- [ ] **텍스트만 primary 색**(탭 활성 등) on 흰 배경은 크기·굵기별로 검증.
- [ ] 대비 수치는 문서에 **“경향·권장 검증”** 수준으로 적고, 최종은 도구 측정에 맡긴다.

### 2.5 오버 엔지니어링

- [ ] 그림자 하나에 **변수 4개**가 꼭 필요한지, 제품 코드에서는 **단일 shadow 토큰**으로 충분한지.
- [ ] Tab/Toggle/Button **타이포 스타일**이 사실상 동일 스케일이면 **공통 토큰**으로 묶을 수 있는가.

---

## 3. 권장 수정사항 운영

- 갱신 시에는 기존 **권장 수정사항**을 먼저 재검토하고, 해결 여부를 확인한다.
- 실제로 해결된 항목은 목록에서 제거하고, 미해결 항목만 유지한다.
- MCP 코드와 불일치(리터럴 `gap`, raw shadow, `text-[16px]` 등)를 **짧은 번호 목록**으로 작성한다.
- **간격**: `gap-[12px]`·`gap-[16px]` 등 **픽셀 리터럴만** 있고 **간격 변수 참조가 없음** → 오토레이아웃 `itemSpacing`·`padding`의 변수 바인딩을 점검한다.
- **색·타이포**: 한 컴포넌트 안에서 **토큰/스타일과 raw hex·고정 px**가 같은 용도에 동시에 쓰이면 SSOT 정리 후보로 기록한다.
- JSON·Figma **이름 충돌**은 반드시 한 항목으로 명시한다.

## 5. 요약 플로우

1. 통합 표와 코드 토큰(`design_tokens.json`)을 대조한다.
2. 체크리스트(§2)로 구조·의미·중복·대비를 점검한다.
3. 권장 수정사항을 피드백 문서에 기록하고 우선순위를 정한다.
4. 해결된 항목은 제거하고, 미해결 항목만 유지한다.
