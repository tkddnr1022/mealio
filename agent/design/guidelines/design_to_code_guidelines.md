# Design to Code 가이드

Figma 디자인을 클라이언트 코드로 옮길 때의 **공통 원칙**과 **변수·스타일·컴포넌트를 코드에 연결하는 방법**을 정리한다.  
**무엇을** 구현할지는 `../../frontend/spec/frontend_architecture_spec.md` 등 명세를 따르고, 본 문서는 **절차·판단 기준**에 초점을 둔다.

**수치·팔레트·타이포 크기·간격·컴포넌트별 필드 이름·비율 등 실제 스펙은 `design_tokens.json`과 `client/app/globals.css`가 단일 근거**이며, 본 가이드에는 넣지 않는다.

---

## 1. 총칙

1. **토큰 우선**  
   픽셀·hex·font-size를 컴포넌트 JSX에 직접 넣지 않는다. 필요한 값이 있으면 **`design_tokens.json`에 정의되어 있는지 확인**하고, 없으면 토큰을 추가한 뒤 **`globals.css`의 `:root` / `@theme`과 동기화**한다.

2. **시맨틱 HTML + 전역 타이포**  
   Figma **텍스트 스타일**은 **시맨틱 태그** 또는 **`globals.css`에 약속된 유틸 클래스**로 반영한다. 어떤 스타일이 어떤 태그·클래스·CSS 변수에 대응하는지는 **`globals.css` 주석과 `design_tokens.json`의 `typography`**를 본다.

3. **컴포넌트 variant는 Figma와 맞출 것**  
   Figma **컴포넌트 프로퍼티**와 **배리언트 값**을 코드의 `variant`(또는 동등한 prop) 이름·의미와 가능한 한 **1:1 대응**시키고, 배리언트 **전체 목록**은 단일 변형 노드가 아니라 **컴포넌트 세트(부모 프레임)** 기준으로 파악한다.

4. **MCP 출력은 참고용**  
   Figma MCP가 주는 React+Tailwind 스니펫은 **그대로 복사하지 않는다.** 프로젝트 스택(Next.js, 기존 토큰·유틸)에 맞게 변환한다.

5. **동일 패밀리, 다른 레이아웃**  
   화면·맥락에 따라 같은 종류의 UI(예: 카드)라도 **비율·간격 등이 다를 수 있다.** 그 경우 **하나의 토큰 블록에 몰아넣지 말고**, `design_tokens.json`의 `component` 등에서 **용도별 블록을 분리**해 두었는지 확인하고, 구현 시 **해당 화면에 맞는 블록**만 따른다. (이름과 필드는 항상 토큰 파일을 본다.)

---

## 2. Figma 변수 (Variables)

### 디자인 측

- **컬러·간격·radius·타이포 숫자** 등은 Figma **Variables**(또는 라이브러리 변수)로 두고, 컴포넌트/텍스트에 **바인딩**하는 것을 권장한다.
- 변수 **이름은 역할을 드러내는 규칙**으로 짓고, 저장소의 `design_tokens.json`과 **같은 의미 축**으로 맞추면 양쪽 동기화가 쉽다.

### 코드 측

- 저장소에서는 **`design_tokens.json`을 우선 근거**로 삼고, Figma와 값이 어긋나면 **디자인 확정 후 양쪽**을 갱신한다.
- CSS/Tailwind 쪽 매핑 패턴(변수 네이밍, `@theme` 연동)은 **프로젝트에 이미 적용된 `globals.css`를 따른다.**

---

## 3. Figma 스타일 (Text / Color styles)

### 텍스트 스타일

- Figma **Text style**이 붙은 텍스트는 **`design_tokens.json`의 `typography`와 `globals.css`의 전역 타이포 규칙**에 맞춰 구현한다.
- **하지 말 것**: 제목·본문·캡션에 **임의의 px·font-weight·line-height·hex를 컴포넌트 파일에 직접 박는 것.** 레이아웃·말줄임·정렬 등 **비타이포 목적의 클래스**만 컴포넌트에 둔다.

### 컬러 스타일

- Figma **Color style**은 **`design_tokens.json`의 시맨틱 컬러**와 대응시키고, UI에서는 **토큰에서 노출된 색 유틸 또는 CSS 변수**만 사용한다. 구체적 토큰 키·유틸 이름은 토큰 파일과 Tailwind 테마 정의를 본다.

---

## 4. Figma 컴포넌트 (Components & Variants)

### 컴포넌트 세트와 프로퍼티

- **컴포넌트 세트(Component set)** 안의 각 배리언트는 레이어 이름이 `PropertyName=Value` 형태로 표시되는 경우가 많다.
- 코드의 `variant` 타입(유니온)은 이 **프로퍼티 값 조합**을 반영한다.  
  (예: 어떤 Navbar는 Figma의 `AdditionalButtons` 프로퍼티 값 세트와 코드 `variant`를 맞춘다.)

### 링크·노드 ID (에이전트·리뷰용)

- **배리언트 하나만** 가리키는 `node-id`로는 **다른 배리언트 목록**이 보이지 않는다.
- **전체 배리언트**를 파악하려면 **컴포넌트 세트를 감싼 프레임** 등 **부모 노드**의 URL을 사용하거나, Figma에서 컴포넌트 세트를 직접 선택해 Variants 패널을 본다.

### Figma MCP 활용 (읽기)

- **`get_metadata`**: 구조·자식 심볼 이름을 빠르게 볼 때 유리. 배리언트 **전체**는 부모 프레임 id로 호출하는 것이 좋다.
- **`get_design_context`**: 구현 참고용 코드·스크린샷. **반드시** 프로젝트 컨벤션에 맞게 재작성한다.

### Code Connect

- Figma 컴포넌트와 저장소 컴포넌트를 매핑할 때는 **Code Connect**를 사용할 수 있다. 도구·워크플로는 Figma MCP의 Code Connect 관련 도구 및 프로젝트 스킬을 따른다.

## 5. 구현 체크리스트 (새 UI 조각)

- [ ] 필요한 값이 **`design_tokens.json`에 정의**되어 있는가? 없으면 **추가 후** `globals.css`와 맞췄는가?
- [ ] 타이포는 **`globals.css` 전역 규칙**(시맨틱 태그·약속 클래스)을 쓰는가?
- [ ] **같은 UI 패밀리라도** 화면에 따라 스펙이 다르면, **`component` 등 용도별 토큰 블록**을 올바르게 골랐는가? (구체적 이름은 토큰 파일 확인)
- [ ] Figma **변수/스타일**과 저장소 토큰이 **의미·값** 면에서 어긋나 있지 않은가?
- [ ] 컴포넌트 **배리언트 전부**가 타입·스토리(또는 문서)에 반영되었는가?
- [ ] 터치 타깃·대비 등 **`design_principle.json`**을 만족하는가?

## 6. 예시: Navbar (요약)

- Figma **컴포넌트 프로퍼티**(예: 우측 액션 종류)에 맞춰 슬롯을 구성하고, 디자인에 없는 조합은 **코드 쪽 `variant` 설계**로만 표현한다.
- 브랜드 워드마크는 변수 `logo`와 같이 **`Coop`으로 코드에 고정**하고, **Logo/Small**에 대응하는 전역 타이포(`typography-logo-small`)에 맡긴다. Navbar 컴포넌트에는 **말줄임·정렬·간격** 같은 레이아웃만 둔다.

이 패턴을 다른 레이아웃·카드·폼 컴포넌트에도 동일하게 적용한다.
